## هسته دادهای پلتفرم پادکست

EchoVerse یک پلتفرم بزرگ انتشار پادکست است که روزانه هزاران کانال و اپیزود را مدیریت می‌کند. با افزایش حجم داده‌ها، تیم فنی از شما درخواست کرده است ساختارهای دادهای مناسب برای بخش‌های مختلف سیستم پیادهسازی کند.

## فاز اول : مدیریت کانال‌های تایید شده (درخت جستجوی دودوی خودمتوازن)

EchoVerse در فقط کانال‌هایی اجازه انتشار دارد که تأیید شده باشند. هر کانال دارای یک ChannelID عددی یکتا است. سیستم باید بتواند:

* بررسی کند آیا یک کانال مجاز به انتشار هست یا خیر
* کانال جدید را تایید کند
* تایید یک کانال را لغو کند

**نکته‌ی بسیار مهم (اجباری):** اطلاعات کانال‌ها باید حتماً در یک درخت جستجوی دودوی خودمتوازن ذخیره شود

استفاده از آرایه، لیست، map، set یا هر ساختار آماده ممنوع است

پیادهسازی باید به صورت دستی انجام شود، به این شکل که:

* برای هر رأس (Node) یک کلاس مستقل تعریف شود
* هر رأس شامل داده‌ی کانال و دو اشاره‌گر / ارجاع به فرزند چپ و راست باشد
* یک کلاس جداوله برای درخت (Tree) تعریف شود که عملیات درج، جستجو و متوازن‌سازی را مدیریت کند
* ساختار درخت تنها از طریق ارجاع بین رأس‌ها شکل بگیرد، نه با ذخیره در یک container دیگر

پیادهسازی درخت به عهدی دانشجو است (مانند AVL Tree)

عملیات مورد نیاز:

* insert(ChannelID)
* delete(ChannelID)
* search(ChannelID)
* display

هدف این فاز، آشنایی با درخت‌های جستجوی دودویی و عملیات تعادلدهی است.

**فاز دوم:** پیشنهاد مسیر گوش دادن بین اپیزودها (گراف وزن دار + دایکسترا)

برای کاربران خود، مسیر پیشنهادی گوش دادن به اپیزودها ارائه می‌دهد. EchoVerse

هر اپیزود یک رأس (Vertex) در گراف است.
وزن یال‌ها نشان‌دهندهٔ میزان شباهت موضوعی یا هزینه‌ی جابجایی ذهنی بین دو اپیزود است
گراف به صورت وزن دار و بدون جهت داده می‌شود

وظایف:

1. گراف وزن دار را بر اساس ورودی بسازید
2. با استفاده از الگوریتم دایکسترا:
    * کوتاه‌ترین مسیر از یک اپیزود مشخص (مبدأ) به سایر اپیزودها را محاسبه کنید
    * مسیر نهایی و هزینه‌ی آن را چاپ کنید

استفاده از الگوریتم Dijkstra اجباری است و استفاده از الگوریتم‌های آماده یا کتابخانه‌های گراف ممنوع می‌باشد.

فاز سوم: ساختار تیم تولید پادکست (درخت + BFS / DFS)

هر پادکست دارای یک تیم تولید با ساختار سلسله‌مراتبی است.

هر عضو می‌تواند چند زیردست داشته باشد

تیم به صورت یک درخت مدل می‌شود

وظایف:

* ساخت درخت تیم تولید
* جستجوی یک فرد خاص با استفاده از DFS یا BFS
* پشتیبانی از:
  * اضافه شدن عضو جدید
  * حذف عضو موجود
  * نمایش درخت

# فاز چهارم (Min-Heap) : صف انتشار اپیزودها

ایپیزودها قبل از انتشار، بر اساس اولویت انتشار مرتب می‌شوند.
عدد کوچکتر = اولویت بالاتر

ایپیزود جدید ممکن است در هر زمان اضافه شود

همیشه اپیزودی که اولویت بالاتری دارد باید زودتر منتشر شود

وظایف:

* Min-Heap پیادهسازی
* عملیات:
  * insert(ID, Priority)
  * extract-min
  * delete
  * display
* Heap Sort مرتبسازی اپیزودها با

قوانین کلی پروژه

* زبانهای مجاز: C++ / Java
* استفاده از کتابخانه‌های آماده ساختمان داده منوع است.
* پیادهسازی تمامی ساختمان داده‌ها به صورت دستی الزامی است.
* پروژه به صورت دو نفره انجام می‌شود.
* مستندسازی و GitHub (Private) الزامی است.
